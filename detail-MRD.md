### 一、Agent 平台核心定位与整体架构
Agent 平台是**智能体统一运行基座与编排中枢**，向下对接大模型、工具、知识与算力，向上承接多端用户请求，核心解决单Agent能力边界有限、多Agent协同混乱、能力复用率低、全链路不可追溯等问题。

整体自下而上分为四层：
1. **基础资源层**：大模型网关、工具集、知识库、算力与存储组件
2. **核心引擎层**：记忆引擎、规划引擎、编排引擎、工具网关、风控引擎
3. **运营管理层**：Agent市场、任务模板、效果评估、低代码配置台
4. **接入交互层**：多端接入网关、会话管理、输出适配

### 二、完整执行链路：从用户输入到任务完成（7大阶段）
#### 阶段1：请求接入与上下文初始化
用户输入（文本/文件/指令）进入平台的第一道关口，完成标准化与前置校验。
1. **多端接入与协议标准化**：统一网关承接API、Web、IM、企业系统等多渠道请求，转换为平台标准任务对象，包含用户身份、会话ID、输入内容、附件、优先级/超时配置等元数据。
2. **前置校验与风控拦截**：完成身份鉴权、配额校验、频率限制，同步做输入内容安全检测，拦截非法与违规请求。
3. **会话上下文加载**：根据会话ID从短期记忆库拉取历史交互、任务断点、用户偏好，补全请求语境，避免用户重复描述。

**核心组件**：接入网关、权限中心、风控引擎、短期记忆库

#### 阶段2：意图理解与任务形式化
将自然语言需求转化为平台可识别、可执行的结构化任务定义。
1. **语义解析与实体抽取**：调用意图理解大模型，识别核心目标、约束条件（时间、格式、成本）、关键实体与输出要求，区分任务类型（闲聊/单步问答/工具调用/复杂长任务）。
2. **任务路由分发**：按复杂度匹配对应链路：
   - 简单问答/闲聊：直接路由通用对话Agent，跳过规划阶段
   - 单工具类任务：进入单Agent执行链路
   - 跨领域多步骤任务：进入任务规划与编排阶段
3. **任务结构化定义**：生成标准Task Schema，包含任务ID、目标描述、优先级、输出格式、依赖资源、重试策略等。

**核心组件**：意图识别模型、任务路由引擎、任务模板库

#### 阶段3：任务规划与多Agent编排
平台核心调度环节，把复杂任务拆解为可执行单元，并匹配最优Agent组队。
1. **拆解决策**：评估任务复杂度，单步可完成的直接指派对应Agent；跨领域、多步骤任务进入正式拆解。
2. **子任务拆解与DAG生成**：由规划大模型（Plan-LLM）将总任务拆解为有明确依赖关系的子任务，生成**有向无环图（DAG）** 执行流，定义每个子任务的输入、输出、先后顺序与并行关系。
3. **Agent智能匹配**：从Agent仓库中，基于子任务能力标签、Agent历史成功率、响应耗时、调用成本，为每个子任务匹配最优Agent（如数据分析Agent、代码Agent、文案Agent）。
4. **协同策略编排**：定义多Agent协作模式（串行流转/并行执行/主从分工/评审博弈），明确角色分工、信息传递规则、异常处理机制。

**核心组件**：任务规划引擎、Agent仓库、DAG编排引擎、全局调度器

#### 阶段4：Agent运行与工具调用执行
任务实际执行阶段，包含Agent内部「思考-行动-观察」循环，以及平台侧的统一管控。
1. **Agent实例化与上下文注入**：平台拉起Agent运行时（容器化/Serverless），注入子任务目标、角色Prompt、相关记忆片段、可用工具权限，完成初始化。
2. **Agent决策循环（ReAct/Reflexion）**：
   - 思考：分析当前子任务进度，判断缺失信息与下一步动作
   - 决策：选择「直接生成内容」「调用工具获取信息」「转交下一个Agent」
   - 观察：获取动作返回结果，更新认知，进入下一轮循环，直到达成子任务目标
3. **统一工具网关代理**：所有工具调用必须经过平台网关，完成参数校验、权限控制、限流熔断、结果标准化，避免Agent直接对接底层工具的混乱与安全风险。
4. **中间状态同步落盘**：每一步执行结果、token消耗、耗时写入执行日志，同步给调度器更新进度，支持长任务断点续跑。

**核心组件**：Agent运行时、工具网关、工具市场、执行日志库、状态管理器

#### 阶段5：结果聚合与质量校验
所有子任务完成后，统一整合输出并做合规与质量兜底。
1. **子任务结果归集**：调度器按DAG依赖收集所有子任务输出，处理异常分支：失败子任务按策略重试/降级，多次失败则触发人工介入。
2. **结果整合与逻辑连贯**：由主控Agent或汇总Agent对多段结果进行拼接、去重、逻辑润色，形成符合总目标的完整输出。
3. **多维度质量校验**：
   - 合规校验：内容安全、敏感信息检测
   - 事实校验：对接知识库做事实性核查，降低幻觉
   - 格式校验：匹配用户要求的输出格式（Markdown/JSON/文件等）
   - 约束校验：确认满足时间、篇幅、成本等约束
4. **异常兜底**：校验不通过则打回对应Agent重跑，或进入人工审核队列。

**核心组件**：结果聚合引擎、质量校验引擎、事实核查模块、人工审核队列

#### 阶段6：响应输出与会话收尾
将最终结果返回用户，完成本次交互的状态更新。
1. **输出格式适配**：根据接入渠道，转换为对应展示形式（纯文本、富文本卡片、文件、流式输出、语音等）。
2. **结果返回**：通过接入网关全量或流式返回用户，支持中途中断、追问、需求修改。
3. **会话状态归档**：标记任务最终状态（成功/失败/人工中断），保存全链路数据，更新短期会话记忆。

**核心组件**：输出适配层、流式推送模块、会话管理器

#### 阶段7：记忆沉淀与能力迭代
平台长期优化闭环，让每一次任务转化为可复用的能力资产。
1. **分层记忆存储**：
   - 短期记忆：完整会话存入关系库，有效期内可直接召回
   - 长期记忆：提取用户偏好、领域知识、成功任务方案，向量化后存入向量库，供后续任务精准召回
2. **效果自动评估**：基于结果质量、耗时、token成本、用户反馈，对本次任务、用到的Agent与工具进行多维度评分。
3. **平台能力迭代**：根据评分优化Agent推荐排序、提示词模板、任务拆解策略，沉淀优质方案为通用任务模板，持续提升平台成功率与效率。

**核心组件**：向量数据库、记忆引擎、效果评估模块、资产运营后台

### 三、完整链路示例
以用户需求「调研2026年国内大模型行业融资事件，输出10页行业报告大纲」为例：
1. 接入：Web端输入需求，鉴权通过，加载历史上下文
2. 意图：识别为「行业调研+文档产出」复杂任务，生成结构化Task
3. 规划：拆解为「融资数据采集→企业背景核实→数据分类统计→报告大纲撰写」4个子任务，匹配对应Agent并生成串行DAG
4. 执行：检索Agent抓取融资事件→企查Agent补全工商背景→分析Agent做赛道/轮次/金额统计→文案Agent生成10页大纲
5. 校验：汇总内容，事实核查校验数据准确性，格式校验确认大纲结构
6. 输出：返回Markdown格式大纲
7. 沉淀：调研框架存入长期记忆，更新「行业调研」任务模板

### 四、核心设计原则
1. **管控与执行分离**：平台负责规划、调度、风控、校验，Agent仅聚焦任务执行，职责边界清晰。
2. **全链路可观测**：每一步输入输出、耗时、成本、状态均可追溯，支撑问题排查与效果优化。
3. **能力资产化**：Agent、工具、知识、任务模板全域复用，避免重复建设。
4. **三级容错机制**：内置重试、降级、人工介入兜底，保障复杂任务成功率。


### Agent 核心组件整体架构
Agent 核心运行时采用「1个主控+4大核心模块」内聚架构，职责边界清晰、可独立迭代：
- 主控：推理调度引擎（驱动 ReAct/Reflexion 全循环）
- 核心模块1：记忆系统（短/长/瞬时三级记忆 + 召回压缩机制）
- 核心模块2：工具能力系统（标准化工具集 + 网关管控 + 智能选择）
- 核心模块3：上下文管理器（Token 水位管控 + 分级压缩执行）
- 核心模块4：安全校验模块（内容合规 + 权限管控 + 幻觉兜底）

## 一、工具能力系统设计
### 1. 工具标准化封装体系
所有工具遵循统一 Tool Schema 规范，确保大模型可识别、可调用、可溯源，核心包含三部分：
- **元数据层**：工具ID、名称、功能描述、适用场景标签、调用成本、平均耗时、风险等级
- **接口层**：输入参数Schema（类型、必填项、约束、示例）、输出结构、错误码定义
- **实现层**：原子工具、复合工具、Agent工具三级封装
  - 原子工具：单一最小能力单元（网页搜索、SQL查询、代码执行、文件读写）
  - 复合工具：固定流程的原子工具组合（企业工商查询 = API调用 + 数据清洗 + 字段归一）
  - Agent工具：将其他专业Agent作为工具调用，实现跨领域能力复用

### 2. 工具选择与调用机制
- **智能召回**：工具数量>20个时，不全部塞入Prompt。采用「向量语义检索+标签过滤」方式，根据当前任务召回Top-K相关工具再注入上下文，大幅降低Token占用与选错概率。
- **标准化调用链路**：
  1. 模型决策调用工具并生成入参
  2. 统一工具网关执行参数校验、权限校验、限流熔断
  3. 调用执行，超时自动重试（最多2次），失败自动匹配替代工具
  4. 返回结果标准化清洗，去除冗余格式与无关字段
  5. 结果注入上下文，进入下一轮推理
- **容错降级**：工具全不可用时，自动提示模型调整方案，避免任务卡死。

### 3. 安全与权限管控
- 粒度化授权：按用户角色、Agent类型开放工具集，高危工具（代码执行、文件写入）需单独审批
- 沙箱隔离：代码、命令行类工具运行在独立沙箱，禁止访问核心系统资源
- 全链路审计：所有工具调用留痕，支持溯源排查

## 二、长短时记忆机制设计
采用**三级记忆架构**模拟人类记忆规律，平衡信息丰富度与上下文效率。

### 1. 短期记忆（工作记忆）
- **定位**：当前会话的实时上下文，直接位于大模型上下文窗口内，是推理的直接信息源。
- **存储内容（按优先级从高到低）**：
  1. 系统提示词（角色定义、核心规则、输出要求）
  2. 当前任务目标、约束条件
  3. 最近N轮对话交互
  4. 本轮任务的工具调用历史与核心结果
  5. 召回的长期记忆片段
- **存储介质**：内存/会话缓存，随会话结束归档落盘，读取零延迟，受Token窗口严格限制。

### 2. 长期记忆
- **定位**：跨会话、跨任务的持久化知识与经验沉淀，按需召回注入短期记忆。
- **分类存储设计**：

| 记忆类型 | 内容定义 | 存储方式 | 典型示例 |
|---|---|---|---|
| 情景记忆 | 用户交互历史、个性化偏好、过往任务记录 | 向量库 + 关系库（带时间、标签元数据） | 用户习惯输出Markdown、上次调研过新能源行业 |
| 语义记忆 | 领域知识、事实数据、业务规则 | 向量知识库 + 结构化库 | 行业术语定义、公司内部审批规则 |
| 流程记忆 | 任务模板、最佳实践、失败解决方案 | 模板库 + 向量库 | 行业调研报告标准框架、某类报错的修复方案 |

- **写入机制**：
  1. 触发条件：任务完成归档、用户明确标注重要信息、工具返回高价值结果
  2. 预处理：提取核心信息、实体识别、标签分类、重要性评分、生成向量嵌入
  3. 去重校验：语义相似度>阈值的记忆执行更新而非新增，避免冗余
- **召回机制**：
  1. 多路召回：语义向量召回 + 关键词精准匹配 + 时间权重过滤 + 标签匹配
  2. 重排序：按「语义相关性+重要性评分+时间衰减」综合打分，取Top-N
  3. 注入方式：打包为「参考记忆」段落，标注来源与时间，插入上下文指定位置

### 3. 瞬时记忆（思考缓存）
- 定位：单轮推理内的中间思考、草稿数据（如ReAct中间步骤、子任务拆解列表）
- 规则：仅保留在当前推理轮次，不进入多轮对话历史，轮次结束自动丢弃，节省Token。

## 三、记忆膨胀解决方案
记忆膨胀核心矛盾：长期记忆持续增长，导致检索变慢、召回精准度下降、冗余信息挤占Token。从「存储-更新-召回」全链路分层治理：

### 1. 分级存储与生命周期淘汰
- **重要性评分体系**：从「用户标注、任务核心度、访问频次、时效性」四个维度打分，将记忆分为三级：
  - 热记忆（高分）：常驻向量库，优先召回
  - 温记忆（中分）：低频存储，仅强相关时召回
  - 冷记忆（低分）：归档至低成本存储，默认不参与召回
- **自动淘汰规则**：
  - 临时记忆（单次会议纪要、临时数据）设置TTL，到期自动删除
  - 超过N个月未访问的冷记忆，自动归档或清理
  - 过时、失效记忆定期下线

### 2. 记忆蒸馏与压缩
- **增量蒸馏**：定期对同主题、同用户的碎片化记忆做模型蒸馏，合并为浓缩记忆，保留核心事实与结论，丢弃冗余对话细节，压缩比可达80%以上。
- **三级摘要结构**：长周期记忆生成「全局摘要-主题摘要-细节记忆」三级结构，召回时优先返回摘要，需要时再调取细节。
- **程序化转化**：将高频重复的操作流程、解决方案，从情景记忆转化为流程记忆（工具/模板），减少重复存储。

### 3. 语义去重与关联治理
- **写入时去重**：新记忆入库前做语义相似度计算，相似度>0.85则执行「补充更新」，将新信息合并到原有记忆。
- **记忆关联图谱**：建立记忆间的实体关联、主题关联，召回时按关联路径扩散，避免返回大量重复同质信息。
- **分域隔离**：按业务域、用户、场景划分记忆分区，跨域默认不互通，避免无关记忆干扰。

### 4. 召回侧精准限流
- 严格控制单次召回数量：默认Top-5~Top-10，按任务类型动态调整
- 设置相关性阈值：相似度低于阈值直接过滤，宁少勿滥
- 动态配额：根据当前上下文剩余Token空间，动态调整召回数量，避免挤占推理空间

## 四、上下文窗口 Token 阈值与压缩规则
核心原则：**分级预警、按优先级压缩、最小信息损耗、保障核心推理能力**。

### 1. Token 阈值分层（以128K窗口为例）
设置四级水位线，触发对应压缩策略：

| 水位等级 | 占用比例 | Token 示例 | 触发动作 |
|---|---|---|---|
| 安全水位 | ≤70% | ≤89.6K | 正常运行，无压缩 |
| 预警水位 | 70%~85% | 89.6K~108.8K | 触发轻度压缩 |
| 临界水位 | 85%~95% | 108.8K~121.6K | 触发中度压缩 |
| 熔断水位 | ≥95% | ≥121.6K | 触发重度压缩，强制兜底 |

### 2. 分级压缩规则
按「信息损耗从低到高、内容重要性从低到高」顺序执行，优先处理冗余度最高、对当前任务影响最小的内容。

#### 轻度压缩（预警水位，信息损耗<10%）
1. **工具结果裁剪**：移除返回的冗余字段、格式标记、无关元数据，仅保留核心结论与关键数据
2. **思考过程精简**：折叠ReAct循环中的重复思考、无效推理，仅保留「决策+结果」
3. **闲聊内容折叠**：将多轮寒暄、重复确认合并为单句摘要，如「用户确认输出格式为PDF」

#### 中度压缩（临界水位，信息损耗20%~30%）
1. **早期对话摘要化**：将会话前半段历史按主题分段生成摘要，替换原文对话
2. **记忆召回裁剪**：降低长期记忆召回数量，仅保留Top-3最相关条目
3. **系统提示精简**：移除非核心约束、示例，保留核心角色与输出规则
4. **子任务归档**：将已完成子任务的详细过程移出上下文，仅保留最终结论

#### 重度压缩（熔断水位，紧急兜底）
1. **滑动窗口截断**：仅保留最近K轮对话，更早内容全部移出，仅保留1条全局会话摘要
2. **工具历史极简压缩**：仅保留工具调用最终结论，移除入参、出参、报错详情
3. **非核心信息全清空**：移除次要记忆、示例、补充说明，仅保留「核心指令+当前任务+最近对话+关键结果」

### 3. 配套执行原则
- 所有压缩操作全量留痕，标注压缩位置与原内容，支持人工回溯
- 动态配额分配：创作类优先保障历史对话，工具类优先保障结果数据，问答类优先保障知识片段
- 流式预判：每轮推理前预计算Token占用，提前触发压缩，避免推理中途超窗报错

---

Agent平台的成本结构中，**大模型Token成本占比70%~90%**，是成本管控的核心；选型的核心原则是「分层匹配、效果兜底」，不盲目追求最强模型，而是按业务场景、链路环节做精细化匹配，在满足效果要求的前提下实现性价比最优。

## 一、模型选型通用框架与分场景标准
### 1. 五大核心评估维度
选型需从效果、成本、性能、合规、工程五个维度综合打分，而非单一维度决策：
- **效果维度**：任务完成率、推理准确率、幻觉率、长上下文理解能力、领域适配性（代码/多模态/数学）
- **成本维度**：单Token单价、平均单任务消耗、万次任务年化成本
- **性能维度**：首Token延迟、生成速率、并发上限、可用性SLA
- **合规维度**：数据隐私合规、境内外部署、数据是否回流训练、行业资质
- **工程维度**：函数调用支持度、API稳定性、流式输出、错误重试机制、生态完善度

### 2. Agent全链路分环节选型策略
Agent并非单模型跑通全流程，而是按环节分层选型，可在不降低核心效果的前提下降本60%以上：

| 链路环节 | 任务特点 | 选型档位 | 典型模型 | 核心诉求 |
|---|---|---|---|---|
| 意图识别/任务路由 | 简单分类、低复杂度 | 轻量小模型 | GPT-4o Mini、Claude 3 Haiku、开源7B/14B | 低成本、低延迟、高并发 |
| 任务规划/复杂推理 | 多步拆解、逻辑决策 | 强模型 | GPT-4o、Claude 3.5 Sonnet、DeepSeek V3 | 高准确率、低幻觉、强推理 |
| 工具调用/单步执行 | 结果解析、单步操作 | 中等模型 | GPT-4o Mini、Claude 3 Haiku、Qwen Plus | 平衡效果与成本 |
| 结果汇总/格式校验 | 内容整合、规范检查 | 中等模型 | 同上 | 低成本、格式合规 |
| 质量终审/高风险场景 | 最终校验、风险兜底 | 强模型 | 同规划层 | 零容错、效果兜底 |

### 3. 典型业务场景选型标准

| 业务场景 | 核心诉求 | 首选档位 | 选型关键指标 | 成本优先级 |
|---|---|---|---|---|
| 通用客服/FAQ问答 | 低延迟、高频、答案标准 | 轻量模型 | 响应速度、常见问题准确率 | 极高 |
| 文案创作/内容生成 | 流畅度、创意性、格式规范 | 中等模型 | 内容契合度、幻觉率 | 中等 |
| 代码生成/调试排错 | 代码准确率、长上下文、调试能力 | 中强模型 | 代码通过率、上下文理解深度 | 中等 |
| 数据分析/SQL生成 | 数值严谨、逻辑准确、工具适配 | 中等模型 | SQL执行通过率、事实一致性 | 中等 |
| 复杂调研/任务规划 | 深度推理、多步拆解、长文档处理 | 强模型 | 任务完成率、逻辑完整性 | 低（效果优先） |

## 二、代码生成场景Token消耗量测算
代码是Agent场景中Token消耗最高的场景之一，核心原因是代码Token密度高、上下文依赖重、多轮调试累加消耗大。

### 1. 代码场景Token消耗构成
单任务全链路Token分为三部分：
- **固定开销（30%~60%）**：系统Prompt（角色、编码规范、工具描述）、召回的历史代码片段、依赖定义，每次调用必带
- **动态开销（40%~70%）**：用户需求、当前代码上下文、报错栈、工具执行结果、模型输出代码
- **累加开销**：ReAct多轮调试、多次工具调用、反思修正，复杂任务可使总消耗翻倍

### 2. 不同复杂度任务量化参考
以标准代码Agent为例，单轮调用Token消耗参考：

| 任务等级 | 场景示例 | 输入Token | 输出Token | 单轮总消耗 | 多轮调试总消耗 |
|---|---|---|---|---|---|
| 简单 | 单函数编写、单行Bug修复 | 3K ~ 5K | 1K ~ 2K | 4K ~ 7K | 8K ~ 15K（2轮） |
| 中等 | 单模块开发、代码重构、单文件调试 | 10K ~ 20K | 4K ~ 8K | 14K ~ 28K | 40K ~ 80K（3轮） |
| 复杂 | 多文件依赖调试、架构改造、全项目排错 | 30K ~ 90K | 10K ~ 30K | 40K ~ 120K | 120K ~ 300K（3-5轮） |

> 补充参考：1KB纯Python代码约对应300~500 Token；中文需求描述的Token密度是英文的1.5~2倍。

### 3. 影响消耗的关键因素
1. **上下文携带量**：是否传入全量依赖文件、接口定义、第三方库文档，是最大变量
2. **调试轮数**：Bug修复通常2~5轮交互，Token随轮次近似线性增长
3. **工具返回结果**：代码执行日志、检索到的代码片段会全额注入上下文
4. **提示词冗余**：编码规范、工具描述的冗余度直接影响固定开销

### 2. 海外计费核心规则与注意事项
1. **分词计数规则**
   基于BPE分词：英文约4个字符=1Token，中文约1.3~2个汉字=1Token；相同字数下中文Token量是英文的1.5~2倍，对应成本更高。

2. **Prompt缓存折扣**
   Anthropic、OpenAI均支持固定Prompt缓存，**缓存命中的输入Token仅按原价的10%~50%计费**。Agent场景中系统Prompt、工具描述、固定规则占比高，开启缓存可降本30%以上。

3. **超长上下文加价**
   部分模型超过指定长度（如128K）后，超出部分Token单价上浮50%~100%，长上下文任务需额外核算成本。

4. **其他计费细节**
   - 函数调用/工具调用不额外收费，仅按输入输出Token计费
   - 流式输出与普通调用计费规则一致
   - 单次调用Token向上取整计费，高频小请求会有小幅溢价
   - 批量API、微调、专属部署有单独定价体系

### 3. 国内模型计费简述
国内模型通常按**每千Token**人民币计价，同档位价格约为海外的1/3~1/2：
- 强模型档（通义千问Max、文心一言4.0、DeepSeek V3）：输入0.02~0.05元/千Token，输出0.06~0.15元/千Token
- 中等模型档：输入0.005~0.015元/千Token，输出0.01~0.04元/千Token
- 轻量模型档：输入0.001~0.003元/千Token，输出0.002~0.006元/千Token

## 四、成本评估方法与优化体系
### 1. 单任务成本测算公式
```
单任务总成本 = 模型调用成本 + 工具调用成本 + 基础设施成本
模型调用成本 = Σ（各环节输入Token量 × 输入单价 + 输出Token量 × 输出单价）
```
示例：复杂代码调试任务，使用GPT-4o，总输入80K Token，总输出20K Token
模型成本 = (80 × 5 / 1000) + (20 × 15 / 1000) = 0.4 + 0.3 = **0.7美元/次**

### 2. 规模化成本预估核心指标
- 日均任务量、平均单任务Token消耗
- 强弱模型路由占比、缓存命中率
- 高峰并发量、重试失败率
- 工具调用频次与单次工具成本

### 3. 核心降本手段（与核心组件设计联动）
1. **分层路由降本**：按任务复杂度自动路由对应档位模型，简单任务走小模型，可降本60%+
2. **上下文压缩降本**：按Token水位分级裁剪冗余内容，移除无效对话、工具元数据，降本20%~40%
3. **缓存复用降本**：开启Prompt缓存、常见问题结果缓存、工具调用结果缓存，命中后成本骤降
4. **记忆精准降本**：优化记忆召回策略，严格控制召回数量与相关性，避免无效记忆挤占Token
5. **流程优化降本**：限制ReAct最大循环轮次，减少无效反思与重试，避免重复调用
6. **提示词精简降本**：压缩系统Prompt、工具描述，移除冗余示例与非核心规则

## 一、基础资源层（底层底座）
为整个平台提供算力、模型、存储等基础依赖，屏蔽底层异构资源差异，向上提供标准化的资源调用能力。

### 1. 算力资源池
- 职责：提供Agent运行、模型推理、任务调度所需的CPU/GPU算力资源；基于容器编排实现弹性扩缩容；支持Serverless模式的Agent实例按需拉起与销毁，降低闲置成本。
- 核心能力：资源隔离、弹性调度、按量计费、多可用区容灾。

### 2. 模型服务网关
平台所有大模型调用的统一出口，是成本管控与模型选型的核心载体。
- 职责：
  - 统一接入国内外主流大模型与私有化开源模型，标准化调用协议与返回格式，屏蔽底层差异
  - 智能路由：按任务复杂度、成本、延迟要求自动匹配最优模型，支持故障自动降级切换
  - 计量统计：精确统计每次调用的输入/输出Token量、耗时、错误码，生成全量计费明细
  - 密钥与配额管理：统一管理模型服务商密钥，按业务线分配调用配额与流控阈值
  - Prompt缓存：复用固定系统提示词、工具描述，命中缓存的Token享受折扣计费，降低成本

### 3. 存储中间件集群
- 职责：按数据特性匹配最优存储介质，提供全平台数据存储能力。
- 典型组件：
  - 关系型数据库：存储会话、任务、配置、用户、审计日志等结构化数据
  - 向量数据库：存储长期记忆、知识库片段的向量嵌入，支撑语义检索
  - 分布式缓存：存放短期记忆、热点配置、会话状态，实现低延迟读写
  - 对象存储：存放文件附件、归档日志、冷记忆数据等大体积非结构化数据

### 4. 基础中间件
消息队列、分布式锁、全文搜索引擎等通用组件，支撑任务异步解耦、高并发调度与全文检索能力。

## 二、核心引擎层（核心大脑）
平台的核心能力中枢，负责任务的规划、调度、执行与记忆管理，决定Agent的执行效果与效率，所有上层能力均基于该层封装。

### 1. 任务编排与调度引擎
全局任务的「指挥中枢」，负责全链路任务的流转与生命周期管控。
- 职责：
  - 接收上层任务请求，维护任务状态机，管理任务全生命周期（排队、执行、暂停、重试、失败、完成）
  - 解析任务DAG（有向无环图），校验子任务依赖关系，按依赖顺序/并行策略分发子任务
  - 异常处理：超时控制、失败重试、降级兜底、异常分支路由，保障任务成功率
  - 多Agent协同调度：支持串行、并行、主从、评审博弈等多种协作模式的编排执行

### 2. 智能规划引擎
任务的「拆解决策中心」，将自然语言需求转化为可执行的任务路径。
- 职责：
  - 任务复杂度评估，简单任务直接跳过规划环节
  - 调用规划大模型，将复杂任务拆解为有依赖关系的子任务，生成标准DAG执行图
  - 动态重规划：执行中遇到工具失败、信息不足、需求变更时，自动调整任务路径重新拆解
  - 沉淀拆解策略与最佳实践，持续优化规划准确率

### 3. 记忆管理引擎
全平台统一的记忆服务中心，为所有Agent提供标准化的记忆读写与召回能力，对应长短时记忆机制。
- 职责：
  - 三级记忆生命周期管理：瞬时记忆、短期记忆、长期记忆的写入、更新、归档、淘汰
  - 记忆加工：语义去重、蒸馏压缩、实体提取、标签分类、重要性评分、向量化
  - 多路召回与重排序：语义向量召回+关键词匹配+时间权重+相关性打分，返回Top-N最优记忆片段
  - 记忆膨胀治理：冷数据归档、定期蒸馏、生命周期淘汰，避免记忆库无限增长
  - 配合上下文管控，按Token水位动态调整记忆注入量

### 4. 统一工具引擎
全平台工具能力的统一管控入口，所有Agent的工具调用必须经过该引擎。
- 职责：
  - 工具标准化注册：遵循统一Tool Schema规范，支持原子工具、复合工具、Agent型工具的接入与发布
  - 智能工具召回：基于当前任务语义做向量检索，召回Top-K相关工具再注入上下文，降低Token占用与选错概率
  - 调用代理网关：参数校验、权限校验、限流熔断、超时重试，统一管控所有工具调用
  - 沙箱执行：代码执行、命令行、文件写入等高风险工具在隔离沙箱中运行，保障系统安全
  - 结果标准化清洗：去除工具返回的冗余格式、无关字段，格式化后注入Agent上下文

### 5. Agent运行时引擎
单Agent的「执行容器」，驱动Agent完成推理-行动循环。
- 职责：
  - Agent实例生命周期管理：按需实例化、资源分配、状态同步、销毁回收
  - 上下文注入：将角色Prompt、任务目标、记忆片段、可用工具列表注入Agent上下文
  - 推理循环驱动：实现ReAct、Reflexion等经典Agent模式，驱动「思考-决策-行动-观察」循环
  - 上下文Token管控：实时计算Token占用，按水位阈值触发分级压缩策略，保障不超窗口
  - 状态同步：每一步执行结果同步给调度引擎与记忆引擎，落盘执行日志

## 六、典型请求的模块流转路径
以一次复杂任务为例，各模块协作流程：
1. 用户请求经**统一接入网关**鉴权后，由**会话管理服务**加载历史上下文，标准化为平台任务
2. 任务下发至**任务编排与调度引擎**，交由**智能规划引擎**拆解为子任务DAG
3. 调度引擎从**Agent仓库**匹配对应Agent，由**Agent运行时引擎**实例化Agent
4. 运行时从**记忆管理引擎**召回相关记忆，从**统一工具引擎**加载可用工具，注入上下文
5. 运行时驱动Agent推理循环，通过**模型服务网关**调用大模型，通过工具引擎调用外部能力，每步状态同步回调度引擎
6. 所有子任务完成后，**质量评估服务**做结果校验，校验通过后由**输出适配模块**格式化返回
7. 全链路数据写入**可观测体系**，有效经验沉淀至记忆引擎与任务模板中心

## 一、任务复杂度识别机制
任务复杂度识别是平台**分层路由、成本管控、资源匹配**的第一道决策关口，核心目标是「简单任务走最短链路降本、复杂任务走完整编排保质」，避免所有任务都走全量规划流程造成的性能与成本浪费。

### 1. 复杂度三级分级标准
平台将任务统一划分为三个等级，不同等级对应不同的执行链路与模型档位：

| 等级 | 定义 | 典型场景 | 对应执行链路 | 选型档位 |
|---|---|---|---|---|
| L1 简单任务 | 单目标、单步完成、无需外部工具与知识，模型直接回答即可 | 常识问答、格式转换、简单文案润色、信息确认 | 单Agent直出，跳过规划与工具 | 轻量模型 |
| L2 中等任务 | 单领域、需1~3次工具调用/知识召回，单Agent可闭环完成 | 单条数据查询、SQL生成、单函数代码编写、文档摘要 | 单Agent + 工具调用，跳过DAG编排 | 中等模型 |
| L3 复杂任务 | 多目标、跨领域、多步骤依赖，需多次工具调用与多角色协同，输出结构化成果 | 行业深度调研、多文件代码重构、完整方案撰写、多维度数据分析 | 全流程规划 + DAG编排 + 多Agent协同 | 强模型做规划，分任务匹配对应档位 |

### 2. 核心识别维度（量化特征）
识别不依赖单一的文本长度，而是从6个维度综合打分，最终输出复杂度等级与置信度：
- **目标维度**：目标数量、清晰度、是否存在模糊约束与多交付物
- **执行维度**：预估执行步骤数、所需工具数量、是否存在多轮迭代调试需求
- **领域维度**：涉及领域数量、领域跨度大小、是否需要跨专业能力
- **知识维度**：是否依赖外部实时知识、长文档上下文、专属领域知识库
- **风险维度**：容错要求、合规等级、是否需要多重校验兜底
- **上下文维度**：历史会话依赖深度、是否需要承接长周期任务断点

### 3. 两级识别落地流程
采用「规则初筛 + 模型精判」的两级架构，兼顾效率与成本，毫秒级完成识别。

#### 第一级：规则初筛（零成本、毫秒级）
优先用硬规则快速判定，覆盖80%以上的明确场景，无需调用模型：
1. **长度规则**：输入文本<200字、无多目标关键词，初步判定L1
2. **关键词规则**：命中「调研、对比、方案、报告、重构、批量、多维度」等关键词，初步判定L2/L3
3. **显式指令规则**：用户明确要求「分步骤、多人协作、调用多个工具」，直接判定L3
4. **历史匹配规则**：与历史已标注任务语义相似度>0.9，直接复用历史复杂度等级
5. **用户/场景规则**：特定高权限场景、固定复杂任务入口，直接判定L3

#### 第二级：模型精判（低延迟、高准确率）
规则初筛置信度不足的边界场景，调用**轻量小模型**（如GPT-4o Mini、开源7B模型）做分类推理：
1. 输入：用户需求 + 上下文摘要 + 可用能力清单
2. 输出：复杂度等级（L1/L2/L3）+ 置信度 + 关键特征标签（如「需搜索」「跨3个领域」「5步以上」）
3. 兜底策略：置信度<0.7时，默认向上升级一级，优先保障任务成功率

### 4. 动态修正机制
复杂度识别并非一次性判定，执行中可动态升级：
- 触发条件：单Agent执行中工具调用次数>3次仍未完成、连续失败重试、结果校验不通过需扩域
- 处理逻辑：自动将任务升级为更高复杂度等级，触发规划引擎重新拆解，转入对应执行链路
- 数据回流：每次升级修正都回流标注数据，持续优化识别模型与规则

## 二、复杂任务规划生成逻辑
复杂任务规划由**智能规划引擎**承接，核心目标是将模糊的自然语言需求，转化为可调度、可执行、有明确依赖关系的子任务DAG（有向无环图），是多Agent协同的「施工图纸」。

### 1. 规划核心设计原则
1. **原子性原则**：每个子任务为最小执行单元，单个Agent可独立闭环完成，无需二次拆解
2. **可观测原则**：每个子任务有明确的输入、输出、成功判定标准，可独立校验与重试
3. **效率优先原则**：无依赖的子任务尽可能并行，缩短整体执行时长
4. **容错兼容原则**：支持单子任务失败重试、降级替代，不导致全局任务失效
5. **成本可控原则**：优先复用模板与已有能力，避免过度拆解造成的Token与调度浪费

### 2. 两种规划模式（兼顾效率与灵活）
#### 模式1：模板化规划（高频场景首选）
针对行业调研、代码审查、数据报表等高频标准化场景，直接复用**任务模板中心**的预置DAG模板，仅做参数化填充。
- 优势：执行速度快、结果稳定、规划成本为零、效果可预期
- 适用：80%的标准化复杂任务
- 匹配逻辑：基于任务语义标签召回Top-3模板，相似度>0.85则直接启用，替换任务目标、约束条件等参数

#### 模式2：通用智能规划（长尾场景适配）
针对无匹配模板的个性化、创新型任务，调用规划大模型动态生成执行DAG，灵活度高。
- 优势：适配任意长尾需求，支持自定义约束
- 适用：20%的非标复杂任务
- 选型：采用强推理模型（如GPT-4o、Claude 3.5 Sonnet）保障拆解准确率，避免规划错误导致执行返工

### 3. 完整规划生成全流程
#### 步骤1：任务语义形式化
先做需求解构，将自然语言转化为结构化的任务定义，是规划的前置基础：
1. 提取核心要素：总目标、交付物格式、约束条件（时间、篇幅、数据来源要求）、禁止项
2. 明确资源边界：可用Agent能力、工具集、知识库范围、成本上限
3. 输出标准Task Schema，作为后续拆解的唯一输入

#### 步骤2：拆解策略决策
判断规划模式：
1. 检索任务模板库，匹配度达标则走模板化规划，参数填充后直接输出DAG
2. 无匹配模板则进入通用智能拆解，根据任务类型选择拆解方法：
   - 流程型任务（如调研、报告）：自上而下逐层拆解
   - 问题型任务（如排错、方案）：思维树多路径拆解，择优选择
   - 创作型任务（如方案、文案）：先框架后细节，分阶段拆解

#### 步骤3：子任务生成与能力标注
规划大模型按「总目标→子目标→执行单元」逐层拆解，生成的每个子任务包含完整元数据：
- 基础信息：子任务ID、名称、目标描述、成功判定标准
- 输入输出：明确依赖的上游数据、输出的数据格式与字段
- 能力标签：所需领域能力、工具权限、模型档位要求
- 执行配置：最大重试次数、超时时间、失败降级方案

#### 步骤4：依赖关系梳理与DAG构建
梳理子任务间的逻辑与数据依赖，生成标准有向无环图：
1. **依赖分类**
   - 数据依赖：下游子任务依赖上游的输出结果，必须串行
   - 逻辑依赖：业务流程上的先后顺序，无强制数据依赖
   - 无依赖：子任务互相独立，可并行执行
2. **DAG构建规则**
   - 唯一入口、唯一出口，避免孤立节点与悬空节点
   - 严格校验循环依赖，发现则自动修正拆解逻辑
   - 同层级无依赖的子任务归为同一并行批次，由调度引擎并发执行
3. **输出结构**：节点列表 + 边依赖关系 + 批次划分

#### 步骤5：规划自检与优化
生成初稿后，执行多维度校验与优化，避免先天缺陷：
1. **完备性校验**：所有子任务组合是否覆盖总目标的全部要求，是否有遗漏的约束
2. **原子性校验**：是否存在粒度过大、仍需二次拆解的子任务
3. **效率优化**：合并冗余子任务，调整依赖关系扩大并行范围
4. **成本优化**：高成本子任务是否有替代方案，是否可通过降级模型实现
5. **容错校验**：关键节点是否有备份方案，单点失败是否导致全局失效

> 工程上可通过「规划生成→反思校验→修正输出」的Reflexion双轮模式实现，用一次额外的轻量模型调用大幅提升规划准确率。

#### 步骤6：输出标准化执行计划
最终输出可直接被编排调度引擎执行的标准计划，包含：
- 任务总目标与全局配置
- DAG拓扑结构（节点、依赖、并行批次）
- 每个子任务的完整元数据与执行配置
- 全局失败兜底策略与异常分支路由

### 4. 动态重规划机制
规划并非一成不变，执行过程中满足触发条件时，规划引擎会基于当前进度动态调整DAG，保障任务不中断：
#### 触发条件
- 子任务连续失败超过重试阈值，原路径不可行
- 结果校验不通过，且无法在当前子任务内修复，需新增子任务
- 用户中途修改需求、补充约束条件
- 依赖的工具/资源不可用，原方案失效
- 执行中发现新增信息，原规划粒度不足，需进一步拆解

#### 重规划执行逻辑
1. 暂停执行流，快照保存已完成子任务的全部结果与状态
2. 输入：原DAG、已完成节点、失败原因、新增约束/需求
3. 规划引擎增量调整：保留已完成节点，仅修正未执行部分与后续路径
4. 校验新DAG的依赖与完备性，确认无误后恢复调度执行

## 三、完整示例：从识别到规划的全链路
**用户需求**：「帮我调研2026年上半年国内大模型厂商的融资情况，对比头部5家厂商的技术路线和商业化进展，输出15页PPT大纲，所有数据标注官方来源」

### 1. 复杂度识别
- 规则初筛：命中「调研、对比、报告」关键词，涉及融资、技术、商业化多个领域，初步判定L3
- 模型精判：确认跨3个领域、需5步以上执行、多次搜索工具调用，最终判定L3复杂任务，进入规划流程

### 2. 规划生成DAG
1. **子任务1：融资数据采集与厂商筛选**
   - 动作：检索2026上半年大模型融资事件，按金额排序选出头部5家厂商
   - 输出：厂商名单、融资事件明细、来源链接
   - 依赖：无，入口节点
2. **子任务2：头部厂商技术路线调研**
   - 动作：逐个检索5家厂商的官方技术文档、产品参数，整理技术路线
   - 输出：技术路线对比表、来源链接
   - 依赖：子任务1（厂商名单）
3. **子任务3：头部厂商商业化进展调研**
   - 动作：检索5家厂商的客户案例、营收数据、合作动态，整理商业化情况
   - 输出：商业化进展对比表、来源链接
   - 依赖：子任务1（厂商名单），与子任务2并行执行
4. **子任务4：多维度综合对比分析**
   - 动作：整合融资、技术、商业化数据，输出核心结论与对比洞察
   - 输出：分析结论总览
   - 依赖：子任务2、子任务3
5. **子任务5：PPT大纲结构化生成**
   - 动作：按15页要求，将分析结论组织为标准PPT大纲结构
   - 输出：完整PPT大纲
   - 依赖：子任务4
6. **子任务6：数据来源标注与事实校验**
   - 动作：核对所有数据标注对应来源，校验核心事实准确性
   - 输出：最终校验版PPT大纲
   - 依赖：子任务5，出口节点

Agent 平台的异常容错体系遵循**「分层防御、逐级兜底、成本可控、全链路可追溯」**的核心原则，从单步执行到全局任务、从自动修复到人工介入构建多级防线，在保障任务成功率的同时，严格控制重试、重规划带来的额外成本，避免无效消耗。

## 一、前置基础：异常分类与结果校验体系
容错的前提是精准识别异常，明确「什么算失败」。所有异常分为两大类，对应不同的处置策略：**调用型异常**（基础设施/接口报错）和**质量型异常**（执行结果不达标）。

### 1. 异常分级标准
| 异常等级 | 定义 | 典型场景 | 核心处置策略 |
|---|---|---|---|
| 瞬时异常 | 临时性、可自愈的报错，重试即可解决 | 网络超时、模型限流、工具临时5xx、算力不足排队 | 指数退避重试 |
| 业务异常 | 逻辑/权限/参数问题，重试无效，需调整方案 | 参数非法、工具权限不足、模型接口不支持、依赖资源不存在 | 降级换路 / 修正参数 |
| 质量异常 | 调用成功但结果不符合要求 | 幻觉错误、格式错误、逻辑不完整、事实不符、未完成目标 | 重跑修正 / 质量校验打回 |
| 致命异常 | 路径完全不可行，当前方案无法完成任务 | 核心工具永久不可用、需求存在矛盾、超出平台能力边界 | 回滚 + 重规划 / 人工介入 |

### 2. 三级结果校验机制
校验是发现质量异常的核心手段，按「规则优先、模型兜底」的原则分层执行，兼顾成本与准确率。

#### 一级校验：规则化校验（零成本，必执行）
基于硬规则快速校验，覆盖80%的显性问题，毫秒级完成：
- **格式校验**：校验输出是否符合要求格式（JSON/Markdown/指定字段）、必填项是否缺失
- **合规校验**：内容安全检测、敏感词拦截、违规内容识别
- **边界校验**：输出长度、数值范围、字段类型是否符合约束
- **状态校验**：工具调用返回码、模型调用状态是否成功

#### 二级校验：事实一致性校验（低成本，工具类任务必执行）
针对依赖外部信息的结果，做事实对齐，降低幻觉：
- 关键数据与工具返回结果、知识库原文做语义比对，核验一致性
- 数据来源校验：标注的来源链接是否存在、数据是否与来源匹配
- 逻辑自洽校验：结果内部是否存在前后矛盾、逻辑冲突

#### 三级校验：目标完成度校验（中等成本，子任务/全局任务必执行）
调用轻量模型对结果做综合评估，判断是否达成子任务/总任务的目标：
- 校验维度：目标覆盖度、逻辑完整性、约束满足度、输出质量分
- 输出：质量评分（0-100）+ 不达标项清单 + 修正建议
- 阈值规则：评分≥80分通过，60-80分打回重跑，<60分判定失败

> 校验粒度分层：单轮Agent推理校验 → 子任务完成校验 → 全局任务终审校验，越靠后校验越严格。

## 二、第一级防线：分级失败重试机制
重试是解决瞬时异常的首选方案，核心原则是**「可重试才重试、有限次数、指数退避、幂等保障」**，避免无效重试推高成本。

### 1. 重试准入规则
仅瞬时异常允许重试，以下场景禁止重试：
- 参数错误、权限不足、业务逻辑不支持等确定性失败
- 内容违规、输出违法等质量合规问题
- 已达到最大重试次数
- 工具侧明确返回「不可重试」错误码

### 2. 分层重试策略
按执行粒度设置不同的重试参数，越底层重试次数越多、间隔越短：

| 重试粒度 | 适用场景 | 最大重试次数 | 退避策略 | 负责模块 |
|---|---|---|---|---|
| 接口级重试 | 模型/工具/内部服务调用超时、限流 | 3次 | 指数退避（1s/2s/4s） | 模型网关 / 工具网关 |
| 单步执行重试 | Agent单轮推理结果校验不通过、工具返回结果无效 | 2次 | 固定间隔，附带修正提示 | Agent运行时引擎 |
| 子任务级重试 | 子任务整体失败、最终校验不通过 | 2次 | 重置上下文，重新执行子任务 | 任务调度引擎 |

### 3. 重试配套保障
- **幂等性保障**：所有写操作（文件写入、数据提交、工具调用）必须携带幂等键，重复调用不产生副作用；子任务执行前做幂等校验，已成功则直接返回结果。
- **熔断保护**：单工具/模型1分钟内失败率>50%时，触发5分钟熔断，自动切换备用资源，避免雪崩式失败浪费成本。
- **重试上下文修正**：质量异常重试时，必须附带「不达标原因+修正要求」，避免重复犯同样错误，而非原样重跑。

## 三、第二级防线：状态回滚与补偿机制
针对产生了中间副作用的失败场景，通过回滚恢复到执行前的一致状态，避免脏数据残留。

### 1. 回滚分级与适用场景
按影响范围分为三级，优先做最小粒度回滚：
1. **操作级回滚**：单工具调用产生副作用后失败，回滚该操作
   - 示例：文件创建失败则删除已生成的临时文件；数据库写入失败则回滚事务
   - 实现：每个写操作类工具必须配套注册**补偿动作（undo）**，执行失败时自动触发补偿
2. **子任务级回滚**：子任务执行中途失败，清理该子任务产生的所有中间数据、临时文件、缓存，恢复到子任务执行前的状态，不影响上游已完成任务
   - 实现：子任务执行前打状态快照，记录所有变更操作；失败时按逆序执行补偿动作，最终删除快照
3. **全局任务级回滚**：任务彻底失败且无法修复，回滚所有已执行的变更操作，恢复系统初始状态
   - 适用：高风险写操作类任务（如批量数据修改、配置发布）
   - 实现：全局事务协调器，统一管理所有子任务的补偿动作，按依赖逆序执行回滚

### 2. 回滚设计原则
- **读操作默认无副作用**：检索、查询、预览类操作无需回滚，仅清理缓存即可
- **回滚失败告警**：补偿动作执行失败时，立即触发人工告警，避免数据不一致
- **回滚留痕**：所有回滚操作全量记录日志，包含原操作、回滚动作、执行结果，支持审计追溯

## 四、第三级防线：降级与备用路径
重试无效时，优先切换备用方案完成任务，而非直接判定失败。核心是**「主备资源池化、降级可预期、效果可接受」**。

### 1. 资源级降级
针对底层依赖不可用的场景，切换同能力替代资源：
- **模型降级**：主模型调用失败/熔断时，自动切换同档位备用模型；高复杂度任务可临时降级为更强模型保障成功率，简单任务可降级为更便宜模型降本
- **工具降级**：主工具不可用时，自动匹配同功能替代工具；如网页搜索失败切换为站内搜索，主API失败切换为备用数据源
- **Agent降级**：指定Agent不可用时，按能力标签匹配同领域次优Agent承接任务

### 2. 功能级降级
针对核心目标可达成、非核心功能可裁剪的场景，做功能取舍：
- 非核心步骤跳过：如数据可视化生成失败，可降级为仅输出表格数据，保障核心结论交付
- 精度降级：高精度分析失败，可降级为基础统计分析，标注降级说明
- 校验降级：非高风险场景，可跳过三级深度校验，保留基础校验，保障交付时效

### 3. 降级管控规则
- 所有降级操作必须标记降级状态，最终输出中明确告知用户「哪些部分做了降级、原因是什么」
- 高风险任务、零容错场景禁止自动降级，必须人工确认
- 降级触发率纳入效果指标，持续优化主路径稳定性，降低降级占比

## 五、第四级防线：动态重规划机制
当现有执行路径完全不可行（核心依赖失效、多次重试校验不通过、需求变更），触发动态重规划，调整任务DAG路径，这是复杂任务的核心容错能力。

### 1. 重规划触发条件
满足任意一条且未超过重规划次数上限时触发：
- 子任务连续2次重试失败，且无可用降级方案
- 结果校验连续3次不通过，无法在原子任务内修复
- 核心依赖工具/模型永久不可用，原路径完全失效
- 用户中途修改需求、补充约束条件，原规划不匹配
- 执行中发现新的必要信息，原规划粒度不足，需新增子任务

### 2. 两种重规划模式
#### 模式1：增量重规划（90%场景首选）
保留所有已完成子任务的结果与状态，仅对未执行的后续路径重新拆解编排：
- 优势：成本低、速度快，不浪费已完成的工作
- 输入：原DAG、已完成节点列表、失败原因、新增约束
- 执行：仅重生成失败节点及后续的子任务与依赖关系，替换原DAG对应部分

#### 模式2：全量重规划（极端场景使用）
废弃原规划，基于当前进度与最新需求重新完整拆解任务：
- 适用：需求发生根本性变更、原规划逻辑存在重大缺陷
- 限制：单任务最多触发1次全量重规划，且需校验成本上限

### 3. 重规划管控规则
- **次数限制**：单任务增量重规划最多2次，全量重规划最多1次，超限则转入人工介入，避免无限规划消耗Token
- **成本管控**：重规划前预估新增消耗，累计消耗超过初始预估3倍时，禁止自动重规划，触发人工审核
- **版本管理**：每次重规划生成新的DAG版本，保留历史版本，支持回溯对比

## 六、第五级防线：人工介入兜底
所有自动容错机制失效后，转入人工队列，作为最终兜底；同时人工处理结果回流优化系统，形成闭环。

### 1. 人工介入触发场景
- 自动重试、降级、重规划均失败，任务无法自动完成
- 高风险任务（涉及资金、合规、生产变更）终审校验不通过
- 用户主动申诉、要求人工处理
- 系统级故障导致大面积任务异常

### 2. 人工操作能力
- 结果修正：直接修改输出内容，校验通过后交付用户
- 路径调整：手动增删子任务、调整依赖关系、指定执行Agent/工具
- 任务终止：判定任务不可完成，告知用户原因并结算成本
- 问题标注：标记异常类型、根因、优化建议，回流到Badcase库

### 3. 闭环优化
- Badcase定期复盘，优化识别规则、规划策略、校验标准
- 高频失败场景沉淀为模板，补充备用路径，降低后续人工介入率
- 工具/模型稳定性数据纳入选型评分，持续优化资源池

## 七、配套保障机制
### 1. 任务状态机管控
所有任务状态流转由状态机严格管控，避免异常状态：
- 核心状态：排队 → 执行中 → 校验中 → 重试中 → 重规划中 → 降级执行 → 成功 / 失败 / 人工处理
- 状态变更必须记录原因、触发条件、耗时，全链路可追溯

### 2. 全链路成本熔断
为避免异常场景导致成本失控，设置多层成本熔断：
- 单轮调用Token上限、单子任务总消耗上限、单任务全局消耗上限
- 达到阈值时自动暂停任务，触发告警，确认后才可继续执行
- 异常高频失败自动熔断对应资源，防止批量损耗

### 3. 可观测与告警
- 全异常分类统计：失败率、重试成功率、降级率、重规划触发率、人工介入率
- 核心指标异常告警：失败率突增、某工具/模型连续失败、成本突增
- 单任务全链路Trace：可还原每一步执行、校验、重试、回滚的完整细节

## 八、完整异常处理流程示例
以「融资数据采集子任务执行失败」为例，完整容错链路：
1. 子任务执行：检索工具调用返回503超时 → 触发接口级重试（3次指数退避）
2. 重试全部失败，工具触发熔断 → 切换备用检索工具执行（资源降级）
3. 备用工具返回结果，校验发现数据不全、质量评分65分 → 触发单步重试，附带修正提示
4. 重试后质量仍不达标，累计2次子任务重试失败 → 触发子任务回滚，清理中间数据
5. 无可用降级方案 → 触发增量重规划，将「全网检索」拆分为「多站点分别检索+数据聚合」，生成新子任务
6. 新子任务执行成功，校验通过 → 继续推进后续任务，全链路标记异常处理过程
7. 若重规划后仍失败，且达到重规划上限 → 转入人工审核队列，由人工补全数据或终止任务

---

代码场景的上下文理解是代码类Agent的核心能力，**纯向量检索无法适配代码的强语法结构、强依赖关联特性**。工业界成熟方案是采用「**预处理结构化解析 + 多维度多路召回 + 知识图谱关联扩展 + Token感知裁剪**」的四级架构，将向量语义、关键词精确匹配、AST语法结构、调用链关系四类能力深度融合，既保证召回广度，又保障上下文的逻辑完整性。

这套能力在Agent平台架构中，归属于**统一工具引擎**下的「代码检索与解析工具集」，同时支撑代码知识库构建与代码Agent的专属记忆召回。

## 一、前置预处理：代码资产结构化加工
所有代码入库前先经过解析引擎做结构化处理，这是后续所有检索与理解的基础，避免纯文本切片破坏代码语义。

### 1. 多语言AST语法解析
基于统一语法解析框架实现多语言兼容，是所有结构化能力的底座。
- **核心技术选型**：优先采用 Tree-sitter 作为统一解析框架，覆盖Java/Python/Go/JS/TS/C++等主流语言，输出标准化AST语法树；编译型语言可额外结合字节码分析提升准确率。
- **核心提取能力**：
  1. 代码结构单元识别：精准切分函数、类、接口、方法、代码块，保证每个切片语法完整、语义自洽
  2. 元数据提取：函数名、参数列表、返回值、修饰符、异常、注释、文档字符串
  3. 依赖识别：导入的包、引用的类、调用的外部方法、继承/实现关系
- **工程优化**：支持增量解析，仅重新解析变更文件，大项目全量解析后毫秒级更新。

### 2. 代码知识图谱构建
基于AST解析结果，构建代码专属知识图谱，存储实体与关联关系，支撑调用链、依赖链查询。
- **实体节点**：项目、模块、文件、类、接口、函数、变量、第三方依赖
- **关联边**：
  - 结构关系：包含、继承、实现、组合
  - 调用关系：函数调用、方法引用、接口调用
  - 依赖关系：文件依赖、模块依赖、第三方库依赖
- **存储选型**：轻量场景用关系型数据库存储邻接表，复杂关联查询用Neo4j等图数据库

### 3. 多粒度索引并行构建
同一份代码同时构建三类索引，对应三路召回能力：
1. **向量索引**：函数/类级代码片段生成Embedding，存入向量数据库，附带语言、模块、文件等元数据标签
2. **全文索引**：代码文本+注释+标识符写入Elasticsearch，针对代码标识符做专属分词优化
3. **结构化索引**：代码图谱、元数据存入关系库/图数据库，支持结构化条件查询

## 二、多维度检索：三路召回+融合重排
采用「多路并行召回 → 去重融合 → 混合重排」的标准流程，兼顾召回率与精准度。

### 1. 三路召回能力详解
#### （1）向量检索：语义相似召回
解决模糊需求、语义匹配问题，找到功能相似但写法不同的代码。
- **模型选型**：采用代码专属Embedding模型，如BGE-Code、Qwen-Code Embedding、CodeBERT，通用文本模型召回准确率通常低30%以上
- **检索策略**：
  - 基础语义召回：将用户需求转化为向量，召回Top-N语义相似片段
  - 元数据过滤：前置过滤语言、模块、文件范围，缩小检索空间提升准确率
  - 多粒度召回：同时召回函数级、类级、模块级片段，按需选择
- **适用场景**：模糊功能查找、Bug模式匹配、相似代码推荐、代码风格对齐

#### （2）关键词/全文检索：精确符号召回
解决精确标识符匹配问题，是向量检索的重要补充，精准定位已知名称的代码单元。
- **引擎选型**：Elasticsearch / Lucene
- **代码专属优化**：
  - 标识符加权：函数名、类名、变量名精确匹配加权，优先级远高于普通文本
  - 驼峰/下划线分词：适配代码命名规范，支持模糊匹配
  - 正则检索：支持按代码模式正则匹配，如查找所有SQL拼接语句
- **适用场景**：已知函数名/类名定位、特定API调用查找、风险关键字扫描

#### （3）结构化检索：关系与元数据召回
基于代码知识图谱做结构化查询，解决纯文本检索无法实现的关系类查询。
- **典型查询能力**：
  - 查找某个函数的所有上游调用者（调用链向上追溯）
  - 查找某个函数的所有下游依赖（调用链向下展开）
  - 查找某个类的所有子类、所有实现类
  - 筛选指定模块、指定语言、指定作者的代码
- **适用场景**：影响面分析、调用链补全、依赖排查、代码审计

### 2. 融合重排层
三路召回结果统一进入重排层，输出最终最优片段集。
1. **去重归一化**：按文件+函数维度去重，合并同一片段的多路命中记录
2. **混合打分排序**：多维度加权综合打分，权重可按场景动态调整
   - 语义相似度分（向量召回得分）：权重40%
   - 符号匹配分（关键词精确匹配度）：权重30%
   - 结构关联分（与当前上下文的调用链距离）：权重20%
   - 热度分（最近修改、高频访问）：权重10%
3. **代码重排模型优化**：高要求场景可接入代码专属Rerank模型，进一步提升排序精准度
4. **Top-N截断**：按Token配额与相关性阈值筛选最终结果

## 三、检索+解析联动：完整代码上下文理解链路
检索只是定位，真正的上下文理解需要结合代码图谱做关联扩展，形成完整的逻辑上下文，再注入Agent。完整执行链路如下：

### 步骤1：需求解析与召回策略选择
Agent运行时接收代码相关任务后，先解析需求类型，选择对应召回组合：
- 模糊功能类：主用向量检索，辅以关键词检索
- 精确问题定位类：主用关键词+结构化检索，辅以向量检索
- 影响面分析类：主用结构化检索，辅以关键词检索

### 步骤2：初筛召回 + 核心节点定位
多路召回并行执行，定位核心代码片段（如目标函数、目标类），作为后续扩展的锚点。

### 步骤3：调用链关联扩展（核心增值步骤）
以锚点代码为中心，基于代码知识图谱做上下文扩展，补全逻辑依赖，这是纯RAG做不到的核心能力。
- **向上扩展**：召回直接/间接调用该函数的上游代码，理解入参来源、调用场景、业务上下文
- **向下扩展**：召回该函数依赖的下游方法、外部接口、数据结构定义，理解内部逻辑依赖
- **横向扩展**：召回同模块、同类的其他相关函数，补充领域逻辑与编码风格
- **扩展层级控制**：默认向上1层、向下1~2层，避免扩展过度导致Token爆炸，可按任务复杂度动态调整

### 步骤4：Token感知的上下文裁剪
根据当前Agent上下文剩余Token水位，做分级裁剪，优先保留核心信息：
1. **优先保留**：核心锚点函数、直接依赖的接口定义、上游入参关键逻辑
2. **次要保留**：间接调用函数摘要、非核心依赖的函数签名（隐藏实现）
3. **可裁剪**：非相关注释、空行、冗余日志代码、历史版本代码
4. **摘要降级**：Token不足时，将非核心函数的完整实现替换为函数签名+功能摘要，压缩比可达70%以上

### 步骤5：结构化注入上下文
将最终整理的代码上下文，按「核心代码 → 依赖定义 → 调用链说明」的结构化格式注入Agent上下文，并标注文件路径、函数名等元数据，方便模型理解与引用。

## 四、典型场景落地示例
**需求**：帮我排查下单接口的空指针异常，给出修复方案
1. **召回定位**：关键词检索「下单、createOrder」定位核心接口函数，向量检索匹配空指针相关异常模式
2. **调用链扩展**：向下扩展该函数调用的所有内部方法，向上追溯入参来源，补全参数传递链路
3. **上下文裁剪**：保留核心链路的完整代码，非核心依赖仅保留函数签名，控制在Token阈值内
4. **注入Agent**：结构化输出「接口入口→参数校验→业务处理→数据库操作」全链路代码，附带调用关系说明
5. **Agent推理**：基于完整上下文定位空指针触发点，生成符合项目编码规范的修复代码

## 五、工程化关键优化点
1. **增量更新**：代码仓库提交后，触发增量解析与索引更新，无需全量重构，保障代码时效性
2. **编译信息增强**：编译型语言结合编译产物（字节码、IR）做调用链分析，比纯AST更精准，可识别反射、动态代理等调用
3. **缓存机制**：高频查询的调用链、常用代码片段做缓存，降低重复解析与检索开销
4. **权限隔离**：按项目、模块粒度做检索权限控制，匹配企业代码权限体系
5. **Badcase回流**：召回错误、上下文缺失的场景回流标注，优化Embedding模型与重排策略

## 一、工具调用管控整体架构
工具调用管控是Agent平台的核心安全与成本闸门，遵循**「统一入口、权责分离、全链路可观测、分级安全兜底」**原则，所有Agent禁止直连底层工具，必须通过统一工具网关完成全流程管控，从注册、发现、调用、执行到审计形成完整闭环。

### 1. 四层管控架构
自顶向下分为资产层、管控层、执行层、观测层，职责边界清晰：

| 层级 | 核心定位 | 核心能力 |
|---|---|---|
| 工具资产层 | 能力注册与标准化 | 工具Schema注册、分类标签、版本管理、能力画像 |
| 网关管控层 | 全量请求统一入口 | 鉴权、限流、参数校验、路由、计费、风险拦截 |
| 执行沙箱层 | 工具运行与隔离 | 普通API执行、代码沙箱、文件操作沙箱、内网代理 |
| 观测审计层 | 全链路可追溯 | 调用日志、指标统计、链路追踪、合规审计 |

### 2. 全链路调用标准流程
一次合法的工具调用经过9个标准环节，无任何一环可跳过：
1. **工具召回**：Agent侧基于当前任务语义，从工具库召回Top-K相关工具，注入上下文
2. **调用生成**：大模型生成工具调用指令（工具ID、入参）
3. **网关接入**：调用请求提交至统一工具网关，携带任务ID、AgentID、用户身份
4. **前置校验**：依次完成身份鉴权、权限校验、参数合法性校验、限流配额校验
5. **路由分发**：按工具类型、负载、可用状态路由到对应执行节点
6. **隔离执行**：对应沙箱/执行器运行工具，超时自动中断
7. **结果处理**：返回结果标准化清洗、冗余裁剪、格式归一
8. **结果返回**：清洗后结果返回Agent运行时，注入上下文
9. **落盘审计**：全量调用日志、参数、结果、耗时、成本写入审计库与计费系统

## 二、权限边界管控体系
核心原则是**「风险分级、最小权限、按需授权、全程留痕」**，从工具风险等级与主体权限两个维度双向约束，杜绝越权调用与高危操作失控。

### 1. 工具风险三级分级标准
按操作副作用、数据敏感程度、安全风险划分为三级，对应不同的授权与执行要求：

| 风险等级 | 定义 | 典型工具 | 执行环境 | 授权要求 |
|---|---|---|---|---|
| R1 低危 | 只读、公开数据、无任何副作用 | 公开网页搜索、公开API查询、格式转换、文本摘要 | 通用执行器 | 全量Agent默认开放 |
| R2 中危 | 只读内部数据、轻度可回滚副作用 | 内部知识库查询、文件读取、非敏感数据查询、预览生成 | 内网代理执行器 | 按业务线申请，绑定指定Agent |
| R3 高危 | 写操作、系统级操作、不可回滚副作用 | 代码执行、文件写入/删除、命令行执行、数据修改、外部系统提交 | 隔离沙箱，资源与网络严格受限 | 单独审批、限时授权、双人复核 |

### 2. 细粒度授权模型
采用「RBAC角色权限 + ABAC属性约束」结合的授权模型，兼顾管理效率与灵活度：
- **主体维度**：按用户角色、Agent类型、业务线、租户划分权限组
- **客体维度**：按工具、工具分组、数据范围、操作类型设置访问边界
- **环境维度**：按调用时间、网络环境、任务风险等级动态调整权限
- **核心规则**：
  - 默认零权限，仅为Agent分配完成任务必需的最小工具集
  - 高危工具与业务场景强绑定，禁止跨场景复用权限
  - 临时权限自动过期，无需人工回收
  - 多租户场景下严格数据隔离，A租户不可调用B租户的内部工具

### 3. 核心管控机制
- **参数白名单**：高危工具的入参设置白名单与正则校验，禁止超出范围的参数传入，比如文件操作限制在指定目录、命令行禁止执行系统命令
- **二次确认**：涉及数据修改、批量操作的高危调用，触发人工二次确认后才可执行
- **操作审计**：全量调用100%留痕，记录调用方、时间、入参、返回结果、耗时、状态，支持全链路追溯，高危操作永久留存
- **越权拦截**：网关实时校验权限，越权调用直接拦截并触发告警，计入安全事件

## 三、Token与成本管控体系
工具调用带来的成本分为两类：**工具自身的调用成本**（第三方API按次/按量收费）和**模型Token成本**（工具描述+返回结果占用上下文Token），其中Token成本占比通常更高，是管控重点。

### 1. Prompt端Token管控：工具描述瘦身
工具描述是上下文固定开销的大头，全量注入可达几十K Token，通过三项机制大幅压缩：
- **按需召回，而非全量注入**：基于任务语义做向量检索，默认仅召回Top 5~8个相关工具注入上下文，工具数量越多，降本效果越显著，通常可降低80%以上的工具描述Token
- **Schema最小化**：工具描述仅保留核心字段（名称、功能、必填参数），移除冗余说明、非必填参数示例、冗余元数据，单工具描述压缩至原长的30%~50%
- **Prompt缓存复用**：固定工具列表、系统规则等静态内容开启模型Prompt缓存，命中后仅按原价10%~50%计费，Agent场景固定开销占比高，缓存命中可降本30%以上

### 2. 返回端Token管控：结果裁剪与降级
工具返回结果是动态Token开销的核心变量，大段冗余结果极易打爆上下文，通过分级裁剪严格控制：
- **标准化字段裁剪**：移除接口返回的冗余元数据、状态码、格式标记、非业务字段，仅保留核心业务数据，通用场景压缩比可达60%以上
- **长度阈值控制**：单工具返回设置Token上限，默认8K Token，超长自动截断并生成摘要，标注「结果已精简」
- **结构化摘要降级**：Token水位紧张时，将长文本结果替换为结构化要点摘要，仅保留关键结论与数据，压缩比可达80%
- **分页增量返回**：支持大数据量工具分页调用，按需拉取，避免一次返回全量数据

### 3. 调用次数与自身成本管控
- **调用次数熔断**：单任务设置最大工具调用次数上限（默认10次），超限自动暂停，避免无限循环与无效重试
- **重复调用缓存**：相同工具+相同入参的调用，结果缓存复用，缓存有效期内直接返回，既节省工具调用费用，又避免重复Token消耗
- **工具成本分级路由**：同功能工具按成本排序，优先调用低成本工具；高成本工具调用需单独校验配额
- **失败重试限次**：瞬时异常最多重试3次，连续失败触发熔断，禁止无意义重试推高成本

### 4. 全局成本熔断机制
- 单工具日/月调用额度、单业务线工具总成本、单任务工具+Token总成本分别设置阈值
- 达到预警线触发告警，达到熔断线自动拦截新调用，避免成本失控

## 四、业务效果量化指标体系
从质量、价值、成本、合规、性能五个维度建立量化指标，客观评估工具调用的业务贡献与运行健康度。

### 1. 调用质量类
衡量工具调用本身的可靠性与准确性
- **调用成功率**：接口正常返回的调用次数 / 总调用次数，反映工具自身稳定性
- **参数错误率**：因模型生成参数非法导致的失败次数 / 总调用次数，反映工具描述清晰度与模型能力
- **工具选错率**：调用后经校验判定与任务目标不匹配的次数 / 总调用次数，反映召回与模型决策准确率
- **重试成功率**：重试后成功的次数 / 总重试次数，反映重试机制的有效性

### 2. 业务价值类
衡量工具对任务效果的实际贡献
- **任务完成率提升**：（有工具时任务完成率 - 无工具时任务完成率）/ 无工具时任务完成率
- **结果准确率提升**：工具辅助下的结果准确率 vs 纯模型准确率的差值
- **人工介入率下降**：工具接入后人工介入占比的下降幅度
- **平均任务耗时缩短**：工具接入后单任务平均时长的下降比例

### 3. 成本效率类
衡量投入产出比与成本管控效果
- **单次调用平均成本**：工具调用费用 + 对应Token成本的均值
- **工具相关Token占比**：（工具描述Token + 工具返回结果Token）/ 任务总Token消耗
- **缓存命中率**：命中结果缓存的调用次数 / 总调用次数，反映缓存优化效果
- **单位任务工具成本**：单任务平均工具开销，用于横向对比与预算核算

### 4. 安全合规类
衡量权限管控与风险防控效果
- **越权调用拦截率**：成功拦截的越权请求 / 总越权请求
- **高危工具审批合规率**：合规审批的高危调用 / 总高危调用
- **违规调用事件数**：月度违规调用的发生次数
- **审计覆盖率**：全量调用的日志留存比例，目标100%

### 5. 性能体验类
衡量调用速度与用户体验
- **平均调用耗时**：从请求发起到结果返回的平均时长
- **P95调用耗时**：95分位耗时，反映长尾性能
- **工具可用率**：月度工具正常服务时长占比
- **并发承载能力**：峰值可支撑的每秒调用次数

## 五、配套兜底与优化闭环
### 1. 熔断降级与备用路由
- **工具熔断**：单工具1分钟内失败率>50%时，触发5分钟熔断，自动切换同功能备用工具，避免雪崩
- **功能降级**：核心工具不可用时，自动降级为替代方案，比如精准搜索失败降级为通用搜索，标注降级说明
- **超时中断**：所有工具调用设置超时阈值，默认30s，超时自动终止并返回超时错误，避免任务卡死

### 2. 效果回流与持续优化
- 调用成功/失败、结果质量评分、用户反馈等数据回流工具画像库
- 持续优化工具召回排序：效果好、成功率高的工具提升权重，选错率高的工具优化描述或降低权重
- 高频失败场景分析：针对参数错误、工具选错等问题，优化工具Schema、补充示例、调整召回策略
- Badcase定期复盘，沉淀为工具设计与管控规则，持续提升调用准确率与稳定性

Agent 平台的幻觉治理是**全链路体系化工程**，不能仅依赖模型自身能力或单一Prompt技巧，核心思路是「前置防控降低概率 → 过程锚定减少编造 → 后置校验拦截漏网 → 长效闭环持续优化」，覆盖模型、Prompt、工具、记忆、规划、输出全环节，同时按业务风险分级管控，在效果、成本、体验之间取得平衡。

## 一、Agent 场景的幻觉核心分类
Agent 相比普通对话场景，幻觉类型更丰富、影响面更大，除通用的事实性幻觉外，还存在三类专属幻觉：
1. **事实性幻觉**：编造不存在的数据、事件、引用来源、数据出处，是最高发也最影响可信度的类型
2. **工具调用幻觉**：分为工具选择幻觉（选错/虚构工具）和参数调用幻觉（参数非法、语义误用、虚构返回结果），是Agent场景最高危的风险点
3. **规划与逻辑幻觉**：任务拆解遗漏、依赖关系错误、推导前后矛盾、执行路径不可行，错误规划会导致后续全链路偏离
4. **记忆幻觉**：检索无关记忆、虚构历史对话、混淆不同会话信息、错误更新记忆，属于「记错了」的系统性偏差
5. **格式幻觉**：输出不符合约定的结构化格式，字段缺失、结构错乱，影响下游系统解析

## 二、六层全链路治理架构
### 第一层：模型选型与前置约束（源头防控）
对应平台模块：模型服务网关、Agent运行时引擎
从生成根源降低幻觉发生概率，是成本最低的治理手段。
1. **分层模型路由**
   - 高风险环节（任务规划、结果终审、高价值输出）选用低幻觉率的强模型，如GPT-4o、Claude 3.5 Sonnet，这类模型的事实一致性显著优于轻量模型
   - 低风险环节（格式转换、简单摘要）可使用轻量模型，通过后续校验兜底
   - 事实密集型场景优先选用具备引用溯源、长上下文保真能力的模型
2. **强约束系统提示词**
   - 基础禁令：明确「未知信息必须如实告知，禁止编造；所有事实断言必须标注依据来源」
   - 推理范式：强制遵循「先列已知依据 → 推导过程 → 输出结论」的思维链结构，禁止跳步输出
   - 边界明确：清晰告知可用工具、知识库范围，禁止超出能力范围生成确定性结论
3. **生成参数管控**
   - 事实类、工具调用类任务强制低温度参数（temperature ≤ 0.3），降低随机性
   - 创作类任务可适度放宽，但设置上限（≤ 0.7），避免失控发散

### 第二层：推理过程自校验（生成中主动抑制）
对应平台模块：Agent运行时引擎
通过分步推理与自我反思，让模型在生成过程中主动纠错。
1. **分步推理 + 中间自检**
   复杂任务拆分为单步执行，每完成一步先做事实与逻辑自检，通过后再进入下一步，避免大段一次性输出的累积幻觉。例如代码生成遵循「签名设计→逻辑实现→边界校验」三步法。
2. **反思修正机制（Reflexion）**
   初稿生成后触发二次推理，要求模型从「事实依据、逻辑自洽、格式合规」三个维度自我审查，列出问题点并修正；高风险场景可执行多轮反思，直到达标或达到最大轮次。
3. **来源强制标注**
   要求所有事实性内容必须标注对应来源ID（知识库片段、工具调用、记忆条目），无来源标注的事实断言，直接在后续校验层判定为疑似幻觉。
4. **不确定性主动拒答**
   模型自主评估答案置信度，信息不足、边界模糊时主动输出「无法准确回答」，而非强行编造补全，对应元认知判真的治理思路。

### 第三层：外部知识与工具锚定（事实锚定，核心手段）
对应平台模块：统一工具引擎、记忆管理引擎、知识库服务
这是Agent场景最核心的幻觉治理手段，本质是**把模型从「凭参数记忆回答」变成「基于权威事实回答」**，从根源上解决知识边界与时效性幻觉。
1. **RAG全链路事实约束**
   - 强制召回：事实类任务必须先召回知识库，要求模型仅基于召回内容作答，禁止使用模型训练知识输出确定性结论
   - 不足拒答：召回内容相关性不足、信息不全时，必须输出「信息不足」，禁止自行补充推演
   - 溯源绑定：所有引用内容绑定原文片段ID，支持一键核对来源
   - 库源准入：仅权威可信数据源可入库，定期更新校验，从源头避免脏数据引发的幻觉
2. **工具调用硬管控**
   - 按需召回工具：仅注入当前任务相关的Top-K工具，减少工具选错概率
   - Schema强校验：工具名、参数格式、取值范围由网关硬校验，非法调用直接拦截，返回明确错误倒逼模型修正
   - 结果唯一基准：工具真实返回结果为唯一事实依据，模型不得超出返回范围做过度推演
   - 关键数据交叉验证：高价值核心数据采用多工具、多来源交叉核对，信息一致才采信，不一致则标记存疑
3. **结构化结果注入**
   工具返回、知识库召回的内容做结构化清洗，明确区分「事实数据」「推导说明」「备注信息」，避免模型混淆事实与描述性文本。

### 第四层：多层级输出校验（后置拦截，兜底防漏）
对应平台模块：质量评估服务
构建「规则校验 → 一致性校验 → 综合终审」三级校验漏斗，逐层拦截幻觉，越往后成本越高、准确率越高。
1. **一级：规则化硬校验（零成本，毫秒级）**
   - 格式校验：检查结构化输出的字段完整性、格式合法性，拦截格式幻觉
   - 来源校验：检查事实断言是否标注来源、标注的来源ID是否真实存在
   - 关键字校验：检查是否出现不存在的工具名、接口名称、违禁表述
2. **二级：事实一致性校验（低成本）**
   基于自然语言推理（NLI）或轻量校验模型，比对输出内容与参考资料（知识库、工具返回结果）的语义一致性，识别三类问题：与参考资料矛盾、无依据编造、偷换概念。
   输出一致性得分，按失配程度分级处置：低失配直接采纳，中失配触发交叉验证，高失配阻断输出并启动溯源。
3. **三级：综合质量终审（中等成本，高风险场景启用）**
   调用强模型对输出做全维度评估，覆盖事实准确性、逻辑自洽性、任务完成度；对疑似幻觉内容标记风险等级，高风险直接拦截，中风险返回用户时标注「待验证」提示。

### 第五层：Agent 专属幻觉专项治理
针对Agent独有的记忆、规划、工具场景做专项优化，弥补通用治理方案的盲区。
1. **记忆幻觉治理（记忆管理引擎）**
   - 写入校验：新记忆入库前做事实性校验与语义去重，避免错误信息进入长期记忆
   - 召回带源：召回的记忆片段必须携带时间、来源、上下文标签，禁止模型脱离片段虚构历史
   - 定期蒸馏：长期记忆定期做事实校验与蒸馏，清理过期、错误、重复的记忆片段
   - 分域隔离：不同会话、不同租户记忆严格隔离，避免交叉污染
2. **规划幻觉治理（智能规划引擎）**
   - DAG合法性校验：生成的任务DAG必须通过循环依赖、原子性、完备性三项规则校验，不通过则自动重规划
   - 执行偏差校验：执行中实时比对实际进度与规划路径，出现偏差则触发动态重规划，避免沿错误路径继续
3. **工具调用幻觉治理（统一工具引擎）**
   - 网关前置拦截：工具选择、参数错误在网关层直接拦截，不进入执行环节
   - 结果语义对齐：工具返回结果做结构化摘要，明确核心结论，降低模型误读概率
   - 执行结果强制覆盖：以工具真实执行结果为准，直接注入上下文，不允许模型自行篡改返回内容

### 第六层：长效闭环优化体系
对应平台模块：运营分析与效果评估模块
1. **Badcase全量归集**
   自动拦截的幻觉、用户反馈的错误、人工审核发现的问题，全量标注分类（幻觉类型、触发场景、根因），进入Badcase库。
2. **针对性迭代优化**
   - 高频幻觉场景：优化系统提示词、补充校验规则、新增对应知识库/工具
   - 工具类幻觉：优化工具Schema描述、调整召回策略、补充正反示例
   - 领域性幻觉：补充领域知识库，或做领域级微调
3. **效果持续度量**
   建立幻觉率核心指标，按场景、模型、环节细分，持续追踪优化效果。

## 三、分级治理策略（平衡成本与效果）
根据任务风险等级匹配治理强度，避免过度治理推高成本：
- **高风险场景**（金融数据、法律意见、医疗信息、生产操作）：全六级治理 + 人工终审，零容忍幻觉
- **中风险场景**（行业调研、代码生成、数据分析）：前五级自动治理 + 抽样校验，平衡效果与成本
- **低风险场景**（文案创作、通用闲聊、信息整理）：前置约束 + 基础规则校验，优先保障体验

## 四、核心量化度量指标
- **整体幻觉率**：千次有效调用中，确认存在幻觉的次数占比
- **幻觉拦截率**：自动校验成功拦截的幻觉数 / 总幻觉数
- **事实准确率**：输出内容中事实性描述的准确比例
- **漏检率**：用户反馈/人工审核发现的漏网幻觉 / 总幻觉数
- **治理成本占比**：幻觉治理带来的额外Token/算力成本 / 总任务成本

### 什么是 Agent 静默漂移
Agent 静默漂移是指：**在无显式报错、无主动配置变更的前提下，Agent 的行为模式、输出质量、能力边界、对齐标准随时间或交互次数发生缓慢、持续性的偏离**。因其变化平缓、无即时故障，不会触发常规报错，很容易被忽略，等到发现时往往已经造成批量效果下降或合规风险。

常见漂移分为五类：
1. **行为漂移**：工具调用率下降、规划步数异常、输出风格/长度偏离、拒答率异常波动
2. **效果漂移**：任务完成率、准确率缓慢下滑，幻觉率隐性上升
3. **对齐漂移**：逐步偏离角色设定、合规边界、业务规则，回答越界
4. **记忆漂移**：长期记忆积累错误/过时信息，持续带偏后续推理
5. **模型漂移**：底层大模型版本静默更新，导致能力分布、输出习惯整体偏移

管控体系遵循「基准锚定 → 实时监测 → 自动纠偏 → 长效治理」四层闭环，覆盖从单会话到全平台的所有漂移场景。

## 一、第一层：基准锚定体系——建立漂移的“标尺”
没有稳定的量化基准，就无法识别漂移。这是所有管控的前置基础，对应平台的**Agent仓库与质量评估服务**。

### 1. 三类黄金基准集
为每个Agent建立专属基线，作为漂移判定的唯一标尺：
- **行为基准**：固化核心行为指标的正常区间，包括工具调用率、平均规划步数、单任务平均轮次、输出Token区间、格式合规率、拒答率、重试率等。
- **效果基准**：沉淀覆盖核心场景、边界Case的标准测试用例集，明确任务通过率、准确率、幻觉率的合格阈值与最低红线。
- **对齐基准**：固化角色定位、业务规则、安全禁令的语义锚点，作为一致性判定的标准。

### 2. 全要素版本化管理
所有可能影响Agent行为的配置全部纳入版本管控，禁止无记录的“热修改”，确保漂移发生后可追溯、可回滚：
- 纳入范围：系统Prompt、模型版本、工具Schema、知识库快照、记忆策略、校验规则
- 每个版本绑定对应基准测试报告，新版本必须通过基准回归才可正式发布

### 3. 核心约束固化
从结构上避免核心规则被上下文稀释：
- 系统提示词拆分结构：**核心约束区（固定不可修改）+ 业务配置区（可迭代）**，迭代优化仅允许修改业务区，禁止触碰核心禁令与角色定义
- 核心规则放在上下文最高优先级位置，多轮压缩时优先保留，不随对话轮次被裁剪

## 二、第二层：多维度静默监测体系——早发现
核心是从「单点故障告警」升级为「趋势性偏移检测」，捕捉缓慢变化的异常，对应平台的**全链路可观测体系与质量评估服务**。

### 1. 行为指标监控（统计级漂移）
针对高频、易量化的行为指标做趋势监控，是成本最低的漂移发现手段。
- **核心观测指标**：工具调用率、工具选错率、平均规划步数、输出Token均值、格式合规率、拒答率、单任务重试率
- **检测逻辑**：采用统计过程控制（SPC）思路，不因为单次波动告警，重点关注**连续同向趋势**：
  - 指标偏离基线±20%
  - 连续3个统计周期（如天级）同方向变化
  - 突破历史控制上下限
- **典型场景**：模型静默升级后，工具调用率从65%缓慢降到35%，无报错但效果大幅下降，通过趋势监控可提前3~7天发现。

### 2. 效果质量监控（能力级漂移）
针对效果类隐性下滑，采用「在线抽样 + 定期回归 + 隐性反馈」三重检测。
- **在线抽样评测**：每日自动抽样N个真实任务，用校验模型自动打分（任务完成度、事实准确率、逻辑一致性），与基线做同比环比对比
- **定期全量回归**：周/月度跑全量基准测试集，输出能力偏移报告，主动发现细分场景的能力退化
- **隐性反馈挖掘**：用户追问率、修正率、重复发起率、会话中断率上升，都是效果漂移的先行信号，早于用户明确投诉

### 3. 对齐合规监控（边界级漂移）
针对合规、角色边界的缓慢放宽，做常态化巡检：
- 角色一致性检测：用轻量模型判别输出是否符合角色定位，是否出现越界回答、偏离人设
- 软违规趋势检测：关注边缘合规内容的占比变化，避免合规边界在迭代中被逐步突破

### 4. 记忆质量监控（记忆漂移专属）
记忆漂移是最隐蔽的一类，错误记忆会持续污染推理，且很难定位根因：
- 定期抽检长期记忆的事实正确性、时效性，统计错误记忆占比
- 监控记忆召回相关性得分的趋势变化，下降则说明记忆库冗余或质量退化

## 三、第三层：分级自动纠偏机制——快修复
发现漂移后，按严重程度分级执行自动纠偏，避免影响持续扩大，对应**Agent运行时引擎、任务调度引擎**。

### 1. 会话级轻度漂移：上下文锚定修复
针对多轮对话中，上下文稀释、用户引导导致的单会话行为偏移：
- 每N轮对话（如5轮）自动注入一次核心约束摘要，强化角色与规则，抵消上下文带来的行为偏移
- 上下文压缩/裁剪时，优先保留核心规则、任务目标、工具说明，优先裁剪闲聊、冗余交互等易带偏内容

### 2. 任务级中度漂移：执行链路强制纠偏
针对单任务内出现的行为偏差，如该调用工具未调用、该拆解未拆解：
- 质量校验层直接打回结果，附带明确纠正指令，强制Agent按规范流程执行
- 单任务内连续出现2次同类偏差，自动切换至更严格的Prompt模板，强化约束
- 工具调用幻觉持续出现时，自动降级为工具列表精简模式，减少选择空间，降低选错概率

### 3. 系统级重度漂移：版本回滚止损
针对全量性的效果/行为滑坡，第一时间止损：
- 核心指标（任务完成率、准确率）跌破最低阈值时，自动回滚至上一个稳定版本的Agent配置（Prompt/模型/工具版本）
- 回滚后触发高优告警，人工介入排查根因，修复后再灰度恢复
- 模型侧漂移可快速切换至备用稳定模型，保障业务连续性

### 4. 记忆漂移专项纠偏
- 校验发现的错误记忆自动打「失效」标签，停止召回，避免继续污染
- 过期、时效性记忆自动触发TTL过期，归档退出活跃召回池
- 相关性低的记忆自动降权，减少召回概率，缓解记忆膨胀带来的偏移

## 四、第四层：根因治理与长效闭环——防复发
### 1. 漂移根因定位
每次漂移事件都做标准化归因，从根源解决：
- **模型侧**：大模型版本静默更新、推理策略调整、安全策略变化
- **平台侧**：Prompt迭代副作用、工具Schema变更、知识库内容更新、记忆策略调整
- **数据侧**：用户输入分布变化、业务场景迁移、记忆库污染累积
- 配套变更关联分析：将指标偏移时间点与所有变更事件做匹配，快速定位触发源

### 2. 变更灰度验证机制
绝大多数人为引入的漂移都来自无验证的配置变更，通过灰度流程前置拦截：
- 所有Agent配置变更（Prompt、模型、工具、记忆策略）必须小流量灰度发布
- 灰度期与基线组做AB对比，核心指标无显著退化才可全量放量
- 高风险变更设置7天观察期，持续监控指标趋势，避免短期正常、长期漂移

### 3. 记忆长效治理
- **定期蒸馏校验**：周期性对长期记忆做全量事实校验与去重，清理错误、重复、过时内容，避免“错误越积越多”
- **写入门槛控制**：仅重要性评分达标、事实校验通过的记忆才可进入长期库，拒绝无效/错误信息累积
- **分域隔离**：不同业务、不同租户、不同场景的记忆严格隔离，避免交叉污染

### 4. 模型迭代适配机制
- 模型供应商版本更新前，提前在测试环境跑全量基准集，验证能力偏移，提前适配Prompt与流程
- 保留至少一个稳定版本模型作为兜底，主模型出现漂移时可快速切换兜底
- 核心场景优先选择版本迭代更可控、更新有预告的模型服务商，降低突发漂移风险

## 典型漂移处理示例
**现象**：代码Agent的工具调用率连续3天下降，代码生成准确率同步下滑，无明显报错。
1. **监测发现**：行为指标监控触发告警，工具调用率从72%降至48%，连续3天下滑
2. **自动止损**：先自动回滚至上一稳定Prompt版本，工具调用率快速恢复
3. **根因定位**：关联变更记录，发现底层大模型静默更新了小版本，对工具调用的指令遵循度下降
4. **优化修复**：调整工具调用Prompt，补充示例，强化指令约束
5. **灰度验证**：小流量测试，工具调用率与准确率恢复至基线水平后全量发布
6. **闭环沉淀**：将该场景加入基准测试集，后续版本更新自动校验
